# Emulator ROM Lab Integration Plan

Status: emulator-only integration plan

Date: 2026-06-28

## Purpose

This report defines the next safe integration phase for the Moto One Hyper ForgeWire ROM Lab.

The phase continues from the ROM lab closeout state: the evidence pipeline is built, but firmware, stock images, recovery anchors, and all physical-device actions remain blocked.

This plan is intentionally emulator-only. It does not authorize firmware acquisition, stock image acceptance, rooting, boot image patching, physical-device booting, flashing, sideloading, slot switching, partition mutation, or any other phone-touch action.

## Integration objective

The objective is to turn the existing read-only emulator bridge into a reusable ROM-lab control surface that can later be connected to ForgeLink and ForgeWire Fabric without expanding its authority.

The bridge should remain:

- emulator-only
- read-only
- contract-gated
- mode-based rather than command-based
- serial-restricted to emulator serial patterns
- hostile to arbitrary shell execution
- explicit about blocked physical-device behavior
- useful for evidence capture, lab status checks, and safe inspection

## Non-goals

This phase does not attempt to:

- boot a Moto One Hyper device image
- build a full ROM
- claim device-specific bringup
- accept a stock firmware package
- accept boot.img, recovery.img, vendor_boot.img, vbmeta.img, super images, sparsechunks, modem/radio images, or persist/EFS artifacts
- introduce physical-device command execution
- provide a rescue workflow
- provide arbitrary ADB or fastboot access
- modify emulator state except through pre-approved, non-destructive launch and read-only inspection boundaries

## Proposed layers

| Layer | Responsibility | Authority boundary |
|---|---|---|
| Contract | Defines allowed target class, forbidden behavior, and safety invariants | No direct device authority |
| Operation catalog | Names the allowed inspection modes | No arbitrary commands |
| Protocol | Defines request/response shape for bridge calls | Rejects unknown fields and direct target fields |
| Runner | Loads contract and operation catalog before invoking implementation | Emulator-only selection |
| JSON wrapper | Provides structured responses for higher-level tools | No device mutation |
| ForgeLink adapter stub | Normalizes future control-surface requests into allowed bridge modes | Cannot bypass bridge protocol |
| Evidence/report scripts | Summarize state into committed markdown reports | Metadata only |

## Safe integration increments

### Increment 1 — publish the emulator-only plan

Create committed planning artifacts that describe the bridge as a bounded ROM-lab surface.

Acceptance evidence:

- report exists
- script exists
- README links report
- local validation remains green

### Increment 2 — expand read-only test matrix

Define what each bridge operation must prove without requiring a live physical device.

Acceptance evidence:

- test matrix report exists
- test matrix script exists
- unit tests continue to cover contract, protocol, runner, operation catalog, adapter stub, and no-phone-touch policy

### Increment 3 — describe ForgeWire / ForgeLink bridge shape

Document the adapter boundary between external orchestration and the emulator-only bridge.

Acceptance evidence:

- bridge plan report exists
- bridge plan script exists
- the plan states that external systems may request named operations only
- the plan states that external systems may not pass raw device identifiers, raw shell commands, physical serials, or mutation intents

### Increment 4 — add optional evidence-run capture

Future work may add evidence-run JSON for emulator bridge validation, but only if it can be produced from local validation without touching the phone.

Acceptance evidence:

- evidence run references an existing work item
- provenance remains concrete only for commands actually run
- no physical-device command appears in the evidence

## Request model for future bridge calls

A future orchestration request should be shaped like this conceptually:

```json
{
  "request_id": "string generated by caller",
  "mode": "known_readonly_mode",
  "params": {
    "bounded_parameter": "value"
  }
}
```

It should not contain:

```json
{
  "serial": "physical-or-arbitrary-device-serial",
  "target": "physical-device",
  "command": "raw shell or host command",
  "adb": "raw adb passthrough",
  "fastboot": "raw fastboot passthrough",
  "write": true,
  "mutate": true
}
```

## Safety invariants

- Emulator-only means emulator-only.
- Named modes are the API surface; raw commands are not.
- External orchestration may request inspections, not authority expansion.
- A bridge adapter cannot add behavior that the contract rejects.
- The phone remains out of scope unless a separate current-session physical gate approves one exact action.
- Evidence blocked means no firmware or image may be treated as trusted.

## Done state for this phase

This phase is done when:

- emulator integration planning artifacts are committed
- README points to the new artifacts
- validation is rerun locally
- no new physical-device capability is introduced
- no firmware or extracted image is committed
