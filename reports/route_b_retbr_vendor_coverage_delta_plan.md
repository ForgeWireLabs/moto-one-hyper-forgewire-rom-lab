## Planning baseline

The expectation map records two source-side manifest families:

| Source manifest | Expected entries | Current planning posture |
|---|---:|---|
| `device/motorola/def` | 261 entries, 259 unique | def-specific set is explicitly anchored to `RPFS31.Q1-21-20-5`; compare against RETBR evidence first |
| `device/motorola/sm6150-common` | 984 entries, 952 unique | common paths are expected on a def image, but the common manifest provenance includes related Motorola sm6150 devices |

The WSL evidence phase proved only report-safe family coverage. It did not run a
full manifest-to-filesystem comparator and did not commit raw inventories or
filesystem trees. Therefore this plan classifies coverage by expected family,
not by definitive path count.

## Classification meanings

| Classification | Meaning |
|---|---|
| confirmed-present | Report-safe `vendor_a` or package evidence names the family clearly enough to plan against it |
| likely-present-but-not-yet-enumerated | Package/source evidence strongly suggests the family exists, but committed reports do not enumerate enough paths for a coverage claim |
| missing-from-report-safe-inventory | Expected family is not visible in the committed report-safe inventory; this is not proof of absence from local images |
| channel-sensitive / cannot compare without exact retus firmware | RETBR evidence exists or is likely, but retus `-1-7-3` equivalence cannot be judged |
| not-applicable / generated / source-side only | The item is not expected as a direct copied proprietary payload, or is generated by extract/setup/build logic |
| blocked from conclusion | A per-file or compatibility answer would require prohibited or not-yet-gated work |

## Def-specific planning result

Expected def families appear broadly satisfiable from the report-safe RETBR
`vendor_a` evidence:

- camera/EEPROM/ArcSoft/QTI CHI;
- Cirrus audio and haptics firmware;
- Motorola pop-up camera HAL/service;
- Egistec fingerprint and Motorola biometric wrapper;
- ST21NFC firmware and libraries;
- display, sensor, and device firmware families.

The delta is not "missing blobs" yet. The delta is that current committed
evidence is family-level, not path-level. The next report-only comparator should
map the 261 def entries into family buckets and mark which buckets have
report-safe examples.

## sm6150-common planning result

Expected common families also appear broadly represented:

- graphics, media, DRM/Widevine;
- audio, ADSP/CDSP, DSP services;
- WiFi, Bluetooth, GNSS, thermal, time, perf, and platform services;
- radio/IMS/QCRIL;
- TEE/QSEE/QTEE, keymaster, gatekeeper, Soter, and related trust stack.

The common-tree risk is different from the def-tree risk. The common manifest
records paths expected for a def build, but the original common provenance
includes related Motorola sm6150 devices. A future report-only comparator should
separate "expected on any sm6150-common build" from "confirmed by RETBR
`vendor_a` evidence" and avoid declaring absence unless the committed
report-safe inventory explicitly supports it.

## Channel-sensitive hold list

These families must remain in the channel-sensitive bucket until exact retus
`RPFS31.Q1-21-20-1-7-3` firmware is available or another explicit gate changes
the decision:

- modem/radio and `radio.img` version relationship;
- qcril and qcril database contents;
- IMS libraries, APKs, and carrier configuration;
- CNE/telephony carrier policy;
- TEE, TrustZone, QSEE/QTEE, Soter;
- keymaster, gatekeeper, keystore;
- fingerprint trustlets;
- DSP/CDSP and camera signed firmware.

RETBR `-5` can inform comparison planning for these families, but it cannot
authorize substitution for retus `-1-7-3`.

## Blocked conclusions

The following conclusions are blocked in this phase:

- exact present/missing counts for all 261 def paths;
- exact present/missing counts for all 984 common paths;
- hashes, versions, and sizes for individual blobs;
- duplicate or conflict resolution for the six overlap libraries;
- compatibility of RETBR radio/IMS/TEE/security blobs with retus `-1-7-3`;
- whether any RETBR-derived file may be imported into `vendor/motorola`;
- whether a Lineage build may start.

Answering those would require a future explicit gate, a report-safe comparator,
or exact retus firmware. This plan authorizes none of those actions by itself.
