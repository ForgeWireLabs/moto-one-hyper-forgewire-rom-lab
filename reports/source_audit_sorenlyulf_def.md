## Audit goal 1 — Android era / does it narrow the A10 / ~A14 / A11 gap?

Yes — this is the central finding. `lineage_def.mk:42-45`:

```make
BUILD_FINGERPRINT := "motorola/def_retail/def:11/RPFS31.Q1-21-20-5/1e3de:user/release-keys"
PRODUCT_BUILD_PROP_OVERRIDES += \
    PRODUCT_NAME=def_retail \
    PRIVATE_BUILD_DESC="def_retail-user 11 RPFS31.Q1-21-20-5 1e3de release-keys"
```

| Dimension | Value | vs phone (`RPFS31.Q1-21-20-1-7-3`) |
|---|---|---|
| Vendor/blob base | Android **11**, `RPFS31.Q1-21-20-5` | **Same `RPFS31.Q1-21-20` family**; differs only in patch suffix (`-5` vs `-1-7-3`) |
| ROM framework | LineageOS 20 (Android 13) | newer framework on the A11 vendor base (standard Lineage pattern) |
| `PRODUCT_SHIPPING_API_LEVEL` | 29 (`device.mk:18`) | A10 launch level (device shipped on A10) — informational |

Interpretation: the previously audited def device trees (ludevjhon, AndroidBlobs)
are anchored to **Android 10 `QPF30.104`**, and the common tree (Fraaxius) is
**lineage-21 (~A14)** with no def specificity. sorenlyulf is the only candidate
whose vendor base matches the phone's A11 `RPFS31.Q1-21-20` stock family. It does
**not** make this a stock-A11 ROM — it is an A13 Lineage build *on* an A11 vendor
base — but it resolves the part that actually mattered: the def-specific tree is
aligned to the vendor/blob era the phone actually carries.

## Audit goal 2 — BoardConfig / device makefiles (def / sm6150 alignment)

`BoardConfig.mk`:

```make
-include device/motorola/sm6150-common/BoardConfigCommon.mk   # consumes the common board contract
DEVICE_PATH := device/motorola/def
TARGET_SCREEN_DENSITY := 420
DEVICE_MANIFEST_FILE += $(DEVICE_PATH)/manifest.xml
DEVICE_MATRIX_FILE := $(DEVICE_PATH)/compatibility_matrix.xml
TARGET_KERNEL_CONFIG := vendor/def_defconfig
BOARD_SEPOLICY_DIRS += $(DEVICE_PATH)/sepolicy/vendor
TARGET_MODULE_ALIASES += snd-soc-cs35l41.ko:... snd-soc-cs47l35.ko:... snd-soc-madera.ko:...
```

`device.mk`:

```make
PRODUCT_BUILD_SUPER_PARTITION := false
PRODUCT_SHIPPING_API_LEVEL := 29
PRODUCT_USE_DYNAMIC_PARTITIONS := true
$(call inherit-product, device/motorola/sm6150-common/common.mk)
$(call inherit-product, vendor/motorola/def/def-vendor.mk)   # generated by extract
```

Alignment is strong and consistent with the target baseline:

- inherits the `sm6150-common` board contract directly (rank-2 common tree);
- `PRODUCT_USE_DYNAMIC_PARTITIONS := true` matches the device profile's `super` +
  A/B layout (`rom_lab/device_def/device_profile.json`);
- device-specific Cirrus audio (cs35l41/cs47l35/madera) and ST NFC config;
- Treble manifests present (`manifest.xml`, `compatibility_matrix.xml`).

`compatibility_matrix.xml` declares the def-specific pop-up camera HAL:

```xml
<hal format="hidl" optional="true">
  <name>motorola.hardware.popup</name>
  <version>1.0</version>
  <interface><name>IMotoPopup</name><instance>default</instance></interface>
</hal>
```

## Audit goal 3 — sepolicy presence and maturity

`sepolicy/vendor/` contains 10 files — real device sepolicy, moderate maturity:

```text
attributes  device.te  file.te  file_contexts  hwservice.te  hwservice_contexts
cameraserver.te  hal_camera_default.te  hal_fingerprint_default.te  hal_motpopup_default.te
```

Notable: dedicated `hal_motpopup_default.te` (pop-up camera), `hal_fingerprint_default.te`,
camera policy, and both `file_contexts` and `hwservice_contexts`. This is
materially more device sepolicy than the recovery/TWRP candidates carry.

## Audit goal 4 — proprietary-files and extract scripts (no blobs acquired)

- `proprietary-files.txt`: **261** blob entries (acdb audio calibration, Cirrus
  cs35l41/cs40l20 firmware, qcom + ArcSoft camera libs, EEPROM libs, sensors).
  This is a manifest only — **no blobs are present in the tree and none were
  acquired** by this audit.
- `extract-files.sh` exports `DEVICE=def`, `DEVICE_COMMON=sm6150-common`,
  `VENDOR=motorola`, delegates to the common tree's extractor, and targets
  `vendor/motorola/def/proprietary`. It is a standard Lineage extract flow that
  pulls blobs from a connected device or a stock dump at build-prep time — it
  does not bundle them. `setup-makefiles.sh` generates `def-vendor.mk`.

No blob was downloaded, extracted, or committed. Blob acquisition remains blocked
(no verified firmware, no root, no approval).

## Audit goal 5 — kernel pointer vs the common tree's expectation

`lineage.dependencies` declares the full dependency graph:

```json
device/motorola/sm6150-common   <- android_device_motorola_sm6150-common
kernel/motorola/sm6150          <- android_kernel_motorola_sm6150
hardware/motorola               <- android_hardware_motorola
system/qcom                     <- android_system_qcom
```

This **confirms and tightens the ranked recommendation's rank-3 kernel pointer**:
the common tree's `TARGET_KERNEL_SOURCE := kernel/motorola/sm6150` and this tree's
`android_kernel_motorola_sm6150` dependency agree, and `BoardConfig.mk` names the
exact defconfig `vendor/def_defconfig`. It also adds two dependencies not
previously surfaced: `hardware/motorola` and `system/qcom`.

## Audit goal 7 — does sorenlyulf become the preferred def device tree?

**Yes.** It is the preferred def-specific device-tree candidate because it is the
only audited tree that is simultaneously: (1) def-specific, (2) anchored to the
phone's A11 `RPFS31.Q1-21-20` vendor base, (3) structurally complete (BoardConfig,
makefiles, sepolicy, proprietary manifest, extract scripts, Treble manifests),
(4) explicit about its kernel/common/hardware dependencies, and (5) fresh.

## What this candidate still does not resolve

- It is **LineageOS 20 (A13 framework)**, not a stock-A11 build; "era resolved"
  means the *vendor/blob base* is A11-aligned, not that a stock-A11 ROM exists.
- `RPFS31.Q1-21-20-5` vs the phone's `RPFS31.Q1-21-20-1-7-3`: same `Q1-21-20`
  base, different patch suffix — blob compatibility is likely but **unproven**
  without the exact firmware or a device dump.
- The master blockers from `reports/rom_source_bringup_ranked_recommendation.md`
  are unchanged: no verified firmware, no blobs, no kernel checkout, no verified
  boot/recovery anchor, no approved physical path. sorenlyulf changes *which*
  tree leads, not the firmware blocker.

## Current decision

Promote sorenlyulf/android_device_motorola_def to the preferred def-specific
device-tree candidate (reference-only; not a build input). Re-point the common
tree to the lineage-20 branch of `sm6150-common`. Keep ludevjhon/AndroidBlobs as
A10 fallback/corroboration. The next higher-value question shifts back to
firmware: confirming an exact or `RPFS31.Q1-21-20`-compatible stock package so
the A11 vendor base can actually be populated.
